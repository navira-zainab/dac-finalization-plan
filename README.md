# Documentation as Code (DaC) вЂ” Finalization Plan

> **Source:** [Confluence page](https://expertflow-docs.atlassian.net/wiki/spaces/~63be65f02341bff4fff674c8/pages/1948844069/Documentation+as+Code+DaC+Finalization+Plan+-+ADD)

---

## Context

The team has BMAD 6.3.0 fully installed (57 skills, 6 agent personas) with an empty `docs/` folder and no CI/CD pipeline yet. The goal is a complete Documentation as Code pipeline: docs authored in Git-tracked markdown, reviewed by BMAD agents, and automatically **published to Docusaurus** (product knowledgebase static site) on merge to `main`.

---

## Architecture: Three Layers

| Layer | What | Tools |
|-------|------|-------|
| **Author** | Markdown files in `docs/` (git-tracked) | BMAD agents + Claude Code |
| **Review** | Editorial + adversarial quality gates | BMAD review skills |
| **Publish** | Auto-build + deploy Docusaurus on merge to `main` | Docusaurus + CI/CD (GitHub Actions / Netlify / Vercel) |

---

## Docusaurus Integration Design

Docusaurus reads directly from the same `docs/` folder BMAD writes to.

### `docusaurus.config.js` (key settings)

```js
// docusaurus.config.js
module.exports = {
  title: 'ExpertFlow Docs',
  tagline: 'Product knowledge base',
  url: 'https://docs.expertflow.com',
  baseUrl: '/',
  onBrokenLinks: 'warn',
  onBrokenMarkdownLinks: 'warn',
  presets: [
    [
      'classic',
      {
        docs: {
          path: 'docs',
          routeBasePath: '/',
          sidebarPath: require.resolve('./sidebars.js'),
        },
        blog: false,
      },
    ],
  ],
};
```

### `sidebars.js` вЂ” Auto-generated from BMAD index files

```js
// sidebars.js
module.exports = {
  docs: [{ type: 'autogenerated', dirName: '.' }],
};
```

Docusaurus auto-discovers all `.md` files. BMAD's `/bmad-index-docs` creates `index.md` files that become Docusaurus category pages.

### Frontmatter convention (all BMAD-generated docs must include)

```yaml
---
id: <slug>
title: <Human-readable title>
sidebar_label: <Short label>
sidebar_position: <integer>
tags: [<relevant>, <tags>]
last_updated: <YYYY-MM-DD>
owner: <agent or person>
status: draft | reviewed | published
---
```

Paige (tech-writer agent) must add this frontmatter block to every doc she creates.

## Phase 4: Automation (CI/CD Pipeline)

### Option A вЂ” GitHub Actions (recommended)

File: `.github/workflows/docs-publish.yml`

```yaml
name: Publish Docs

on:
  push:
    branches: [main]
    paths: ['docs/**', 'docusaurus.config.js', 'sidebars.js']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run build
      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build
```

### Option B вЂ” Netlify / Vercel (zero-config)

- Connect the Git repo; set build command: `npm run build`; set publish dir: `build`
- Auto-deploys on every push to `main` вЂ” no YAML needed

### Claude Code Hook вЂ” Auto Sidebar Refresh

File: `.claude/settings.json` (PostToolUse on Write to `docs/`)

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write",
      "hooks": [{
        "type": "command",
        "command": "node scripts/refresh-sidebars.js > /dev/null || true"
      }]
    }]
  }
}
```

This keeps `sidebars.js` in sync after every BMAD agent write without manual intervention.

## Key Decisions

1. **Docusaurus** is the chosen static site generator for the product knowledgebase.
2. **BMAD agents** author docs; **Paige** (tech-writer) is always the final author.
3. **`docs/`** is the single source of truth вЂ” Docusaurus reads directly from it.
4. **Party mode** for cross-functional planning; single agent for authoring.
5. **Shard after review** вЂ” never shard a monolithic draft before adversarial review passes.
6. **`docs/distillates/`** and `docs/project-context.md` are excluded from Docusaurus (AI-internal only).

# Documentation Pipeline Architecture

> **Source:** [Confluence page](https://expertflow-docs.atlassian.net/wiki/spaces/~63be65f02341bff4fff674c8/pages/1950220314/Documentation+Pipeline+Architecture)

---

## Overview

There are **4 distinct pipelines**, each triggered by a different event and serving a different purpose.

---

## Pipeline 1 вЂ” Contribution Validation (PR Gate)

**Trigger:** Every pull request that touches `docs/**`  
**Purpose:** Catch problems before they reach production. Acts as the quality gate.

| Stage | Checks |
|-------|--------|
| **Stage 1: Lint & Format** | markdownlint (heading hierarchy, list style), Prettier (consistent formatting), Frontmatter validation (title, slug present) |
| **Stage 2: Content Quality** | Spell check (cspell with custom dictionary), Internal broken link check, No placeholder text (TODO, FIXME, TBD) |
| **Stage 3: Build Verification** | Docusaurus build (`npm run build`), Fails fast if any page breaks the build |

**Outcome:**
- вњ… All pass в†’ PR ready for human review
- вќЊ Any fail в†’ PR blocked, author notified

---

## Pipeline 2 вЂ” Preview Deploy (PR Review Aid)

**Trigger:** PR opened/updated (runs after Pipeline 1 passes)  
**Purpose:** Reviewers see the rendered Docusaurus output вЂ” not raw Markdown вЂ” before approving.

| Stage | Action |
|-------|--------|
| **Stage 1** | Build Docusaurus (preview mode) |
| **Stage 2** | Deploy to staging environment (e.g. `docs-pr-123.vercel.app`) |
| **Stage 3** | Post preview URL as PR comment |

**Why this matters:** Reviewers catch formatting issues, broken images, and navigation problems that Markdown diffs hide.

---

## Pipeline 3 вЂ” Production Publish (Merge Pipeline)

**Trigger:** Merge to `main` branch  
**Purpose:** The authoritative publish step. Only runs on approved, merged content.

| Stage | Action |
|-------|--------|
| **Stage 1** | Build Docusaurus (production) |
| **Stage 2** | Full link check (internal + external) |
| **Stage 3** | Deploy to production host (e.g. `docs.expertflow.com`) |
| **Stage 4** | Post-deploy smoke test (verify homepage returns 200 OK) |
| **Stage 5** | Notify team via Slack / email вЂ” "Docs updated" |

---

## Pipeline 4 вЂ” Scheduled Maintenance

**Trigger:** Cron schedule (weekly, e.g. every Monday 08:00)  
**Purpose:** Catches rot that no PR introduced вЂ” external links die, content goes stale.

| Stage | Action |
|-------|--------|
| **Stage 1** | External link checker (links to 3rd-party sites, docs, APIs) |
| **Stage 2** | Search index rebuild (if using Algolia DocSearch) |
| **Stage 3** | Stale content report вЂ” flag pages not updated in 6+ months |
| **Stage 4** | Open GitHub Issue automatically if failures are found |

---

## Where BMAD Fits

BMAD-generated drafts in `_bmad-output/` are **not** auto-published. A human promotion step moves them into `docs/` and opens a PR вЂ” that PR then enters Pipeline 1 like any other contribution. This keeps AI-generated content subject to the same review gate as human-written content.

---

## Recommended Tooling Per Stage

| Stage | Tool |
|-------|------|
| Markdown lint | `markdownlint-cli2` |
| Spell check | `cspell` |
| Internal links | `docusaurus build` (catches these natively) |
| External links | `lychee` or `linkinator` |
| Build | `docusaurus build` |
| Preview deploy | Vercel (automatic per-PR previews, zero config) |
| Production deploy | Vercel / GitHub Pages / S3+CloudFront |
| Notifications | GitHub Actions в†’ Slack webhook |
