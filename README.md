# Documentation as Code (DaC) — Finalization Plan

> **Source:** [Confluence page](https://expertflow-docs.atlassian.net/wiki/spaces/~63be65f02341bff4fff674c8/pages/1948844069/Documentation+as+Code+DaC+Finalization+Plan+-+ADD)

---

## Context

The team has BMAD 6.3.0 fully installed (57 skills, 6 agent personas) with an empty `docs/` folder and no CI/CD pipeline yet. The goal is a complete Documentation as Code pipeline: docs authored in Git-tracked markdown, reviewed by BMAD agents, and automatically **published to Docusaurus** (product knowledgebase static site) on merge to `main`.

---

## Architecture: Three Layers

| Layer | What | Tools |
|-------|------|-------|
| **Author** | Markdown files in `docs/` (git-tracked) | BMAD agents + Claude Code |
| **Review** | Editorial + adversarial quality gates | BMAD review skills |
| **Publish** | Auto-build + deploy Docusaurus on merge to `main` | Docusaurus + CI/CD (GitHub Actions / Netlify / Vercel) |

---

## Docusaurus Integration Design

Docusaurus reads directly from the same `docs/` folder BMAD writes to.

### `docusaurus.config.js` (key settings)

```js
// docusaurus.config.js
module.exports = {
  title: 'ExpertFlow Docs',
  tagline: 'Product knowledge base',
  url: 'https://docs.expertflow.com',
  baseUrl: '/',
  onBrokenLinks: 'warn',
  onBrokenMarkdownLinks: 'warn',
  presets: [
    [
      'classic',
      {
        docs: {
          path: 'docs',
          routeBasePath: '/',
          sidebarPath: require.resolve('./sidebars.js'),
        },
        blog: false,
      },
    ],
  ],
};
```

### `sidebars.js` — Auto-generated from BMAD index files

```js
// sidebars.js
module.exports = {
  docs: [{ type: 'autogenerated', dirName: '.' }],
};
```

Docusaurus auto-discovers all `.md` files. BMAD's `/bmad-index-docs` creates `index.md` files that become Docusaurus category pages.

### Frontmatter convention (all BMAD-generated docs must include)

```yaml
---
id: <slug>
title: <Human-readable title>
sidebar_label: <Short label>
sidebar_position: <integer>
tags: [<relevant>, <tags>]
last_updated: <YYYY-MM-DD>
owner: <agent or person>
status: draft | reviewed | published
---
```

Paige (tech-writer agent) must add this frontmatter block to every doc she creates.

```markdown
## Documentation Rules
- All docs live in `docs/` and must include the standard frontmatter block
- Paige (tech-writer) is the final author for all published content
```

## Phase 2: Sprint Documentation Cycle (Recurring)

### Content Creation (Sprint Start)

| Who | Skill | Output |
|-----|-------|--------|
| John (PM) | `/bmad-agent-pm` | `docs/product/prd.md` |
| Mary | `/bmad-agent-analyst` | `docs/product/user-research.md` |
| Sally | `/bmad-agent-ux-designer` | `docs/product/ux-design.md` |
| Winston | `/bmad-agent-architect` | `docs/technical/architecture.md` + ADRs |
| Amelia + Paige | `/bmad-agent-dev` → `/bmad-agent-tech-writer` | `docs/guides/developer-guide.md` |

**Key rule:** Party mode for cross-functional planning, single agent (Paige) for final authoring.

### Review Gates (mandatory before merge)

Run in sequence on each changed doc:

```
/bmad-editorial-review-prose     ← copy-editing pass
/bmad-editorial-review-structure ← structure / information architecture
/bmad-review-adversarial-general ← cynical review
/bmad-review-edge-case-hunter    ← edge cases + boundary conditions
/bmad-checkpoint-preview         ← human sign-off gate
```

For large docs (>500 lines), BEFORE review:

```
/bmad-shard-doc
```

**Rule:** Shard AFTER the adversarial review passes on the monolithic draft, not before.

## Phase 3: Git Branch Strategy

```
main          ← Docusaurus build + deploy triggers here
  └── docs/sprint-{n}    ← sprint branch (merged at sprint end)
        └── docs/feat-{x}  ← BMAD agent feature branches
```

- BMAD agents work on `docs/feat-{x}` branches only
- PRs require `/bmad-checkpoint-preview` sign-off as the human gate
- Sprint branch merges to `main` at sprint end → triggers Docusaurus build + deploy

## Phase 4: Automation (CI/CD Pipeline)

### Option A — GitHub Actions (recommended)

File: `.github/workflows/docs-publish.yml`

```yaml
name: Publish Docs

on:
  push:
    branches: [main]
    paths: ['docs/**', 'docusaurus.config.js', 'sidebars.js']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run build
      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build
```

### Option B — Netlify / Vercel (zero-config)

- Connect the Git repo; set build command: `npm run build`; set publish dir: `build`
- Auto-deploys on every push to `main` — no YAML needed

### Claude Code Hook — Auto Sidebar Refresh

File: `.claude/settings.json` (PostToolUse on Write to `docs/`)

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write",
      "hooks": [{
        "type": "command",
        "command": "node scripts/refresh-sidebars.js > /dev/null || true"
      }]
    }]
  }
}
```

This keeps `sidebars.js` in sync after every BMAD agent write without manual intervention.

```

2. Cross-reference with any Jira issues resolved in that period

```

## Key Decisions

1. **Docusaurus** is the chosen static site generator for the product knowledgebase.
2. **BMAD agents** author docs; **Paige** (tech-writer) is always the final author.
3. **`docs/`** is the single source of truth — Docusaurus reads directly from it.
4. **Party mode** for cross-functional planning; single agent for authoring.
5. **Shard after review** — never shard a monolithic draft before adversarial review passes.
6. **`docs/distillates/`** and `docs/project-context.md` are excluded from Docusaurus (AI-internal only).
